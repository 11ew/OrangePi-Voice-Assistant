## 📋 任务目标

改善 VAD 检测效果，引入 RNNoise 消除底噪，解决 Phase 3 发现的问题：
- Silero VAD 对底噪非常敏感
- 在有持续底噪的环境下，无法正确分割语音片段
- 当前使用基于能量的 VAD 作为替代方案

**预期效果：**
在 VAD 检测之前先消除底噪，让 Silero VAD 能够在有底噪的环境下正常工作。

```python
class RNNoiseDenoiser:
    """RNNoise 降噪器"""

    def __init__(self, sample_rate: int = 48000):
        """初始化（RNNoise 只支持 48kHz）"""

    def process_frame(self, audio_frame: np.ndarray) -> tuple:
        """处理单个音频帧（480 样本 = 10ms）"""

    def process_audio(self, audio_data: np.ndarray) -> tuple:
        """处理完整音频数据"""
```

- 输入/输出格式：float32 数组

## ❌ 发现的问题

### 问题 1：RNNoise 过度降噪

**现象：**
使用 Silero VAD 测试降噪后的音频，**无法检测到任何语音片段**！

```bash
python3 scripts/test_vad_simple.py output/vad_record_test_denoised.wav

📊 检测结果:
   - 总语音片段数: 0  ❌

⚠️  未检测到任何语音片段！
```

**原因分析：**

详细统计对比：
```
原始音频统计:
  最大值: 1432
  平均值: 65.6
  RMS: 114.6

降噪音频统计:
  最大值: 1341
  平均值: 31.8
  RMS: 101.0

信号衰减:
  最大值衰减: 6.4%
  平均值衰减: 51.5%  ⚠️ 关键问题！
```

**关键发现：**
- RNNoise 将平均信号强度降低了 **51.5%**
- 这说明 RNNoise 把大量的语音信号也当作噪音消除了
- 降噪后的音频信号太弱，Silero VAD 无法检测到语音

### 问题 2：RNNoise 的局限性

**RNNoise 的设计目标：**
- 主要用于 **VoIP 通话场景**（如 WebRTC）
- 假设输入是 **清晰的语音 + 背景噪音**
- 优化目标是 **保持语音清晰度**，而不是保持信号强度

**我们的场景：**
- 录音环境：有持续底噪
- 录音质量：信号本身就不强（最大值只有 1432，远低于 32768）
- VAD 需求：需要足够的信号强度来区分语音和静音

**不匹配点：**
1. RNNoise 会降低整体信号强度
2. 我们的录音信号本身就弱
3. 降噪后信号更弱，VAD 无法检测

## 💡 解决方案分析

### 方案 1：增益补偿（不推荐）

**思路：**
降噪后对音频进行增益放大，补偿信号衰减。

```python
# 降噪后增益补偿
denoised_audio = denoised_audio * 2.0  # 放大 2 倍
```

**问题：**
- ❌ 会同时放大残留噪音
- ❌ 可能导致信号失真
- ❌ 无法恢复已经丢失的语音信息

### 方案 2：调整 VAD 阈值（不推荐）

**思路：**
降低 Silero VAD 的检测阈值，适应降噪后的弱信号。

```json
{
  "threshold": 0.1  // 从 0.3 降低到 0.1
}
```

**问题：**
- ❌ 会增加误检率
- ❌ 可能检测到更多噪音
- ❌ 失去了降噪的意义

### 方案 3：选择性降噪（可行但复杂）

**思路：**
只对静音段进行降噪，语音段保持原样。

```python
# 伪代码
if vad_prob > 0.5:  # 语音段
    output = input  # 不降噪
else:  # 静音段
    output = rnnoise_process(input)  # 降噪
```

**问题：**
- ⚠️ 需要先进行 VAD 检测（循环依赖）
- ⚠️ 实现复杂度高
- ⚠️ 可能在语音边界产生不连续

### 方案 4：继续使用基于能量的 VAD（推荐 ⭐）

**思路：**
保持 Phase 3 的解决方案，不引入 RNNoise。

**理由：**
1. ✅ 基于能量的 VAD 已经能够准确分割语音片段
2. ✅ 实现简单，性能高效
3. ✅ 适合我们的录音环境（有底噪）
4. ✅ 不会降低信号强度

**Phase 3 的测试结果：**
```
基于能量的 VAD（能量阈值 100）：
- 成功检测到 3 个语音片段 ✅
- 片段 1: 0.80s - 1.10s (时长 0.30s)
- 片段 2: 3.50s - 3.90s (时长 0.40s)
- 片段 3: 7.20s - 8.10s (时长 0.90s)
```

### 方案 5：改善录音质量（根本解决方案 ⭐⭐）

**思路：**
从源头解决问题，提高录音质量。

**具体措施：**
1. **提高录音音量**
   ```bash
   # 当前音量：10 (8%)
   amixer set Capture 15  # 提高到 15 (12%)
   ```

2. **使用更好的麦克风**
   - 考虑使用外置 USB 麦克风
   - 选择带降噪功能的麦克风

3. **改善录音环境**
   - 减少环境噪音源
   - 使用隔音材料
   - 选择更安静的录音环境

4. **使用硬件降噪**
   - 某些麦克风自带硬件降噪
   - 效果比软件降噪更好

## 📝 技术总结

### RNNoise 的优缺点

**优点：**
- ✅ 基于深度学习，降噪效果理论上很好
- ✅ 专门针对语音优化
- ✅ 开源免费，社区活跃

**缺点：**
- ❌ 会降低整体信号强度（平均衰减 50%+）
- ❌ 对弱信号不友好
- ❌ 固定采样率（48kHz），需要重采样
- ❌ 增加计算开销

### 基于能量的 VAD 的优缺点

**优点：**
- ✅ 实现简单，计算高效
- ✅ 不会降低信号强度
- ✅ 适合有底噪的环境
- ✅ 参数可调，灵活性高

**缺点：**
- ⚠️ 需要手动调整阈值
- ⚠️ 对突发噪音敏感
- ⚠️ 不如深度学习方法智能

### 经验教训

1. **不要盲目追求新技术**
   - RNNoise 虽然先进，但不一定适合我们的场景
   - 简单的方案往往更可靠

2. **从源头解决问题**
   - 提高录音质量比后期处理更重要
   - 硬件投入往往比软件优化更有效

3. **充分测试验证**
   - 理论上可行不代表实际有效
   - 必须用真实数据测试

4. **保持方案简单**
   - KISS 原则：Keep It Simple, Stupid
   - 复杂的方案往往带来更多问题
