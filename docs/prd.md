## 🎯 项目概述

### 项目背景

EdgeTTSdemo 是一个基于 Orange Pi AI Pro (Ascend 310B) 开发板的实时语音助手系统。项目已有基础代码和配置，但各模块功能状态不明确，需要从头开始验证和重建，确保每个模块都能正常工作。

### 当前状态分析

#### ✅ 已有且可用的组件

1. **LLM 引擎 (DeepSeek)** ✅ 已验证
   - 文件：`scripts/deepseek_chat.py`
   - API Key：已配置
   - 状态：测试完成，平均响应 2.188秒

2. **TTS 引擎 (Edge-TTS)** ✅ 已验证
   - 文件：`scripts/tts_generate.py`
   - 功能：文字转语音，支持中英文
   - 状态：测试完成，平均生成 2.7秒

3. **ASR 引擎 (Paraformer)** ✅ 已验证
   - 文件：`scripts/speech_to_text.py`
   - 模型：Paraformer INT8 量化模型
   - 状态：集成完成，识别准确率优秀

## 🔧 技术架构

### 技术栈

| 组件 | 技术选型 | 版本 | 运行环境 |
|------|---------|------|---------|
| **VAD** | Silero VAD (sherpa-onnx) | latest | 本地 CPU |
| **ASR** | Paraformer INT8 | latest | 本地 CPU/NPU |
| **LLM** | DeepSeek API | deepseek-chat | 云端 |
| **TTS** | Edge-TTS | ≥6.1.0 | 云端 |
| **编排** | Python AsyncIO | 3.8+ | 本地 |
| **音频** | ALSA + sounddevice | latest | 本地 |

### 系统架构

```
┌─────────────────────────────────────────────────────────┐
│                  实时语音助手系统                        │
└─────────────────────────────────────────────────────────┘

┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   麦克风输入  │───▶│  音频缓冲区   │───▶│  VAD 检测器  │
│  (sounddevice)│    │  (asyncio)   │    │ (Silero VAD) │
└──────────────┘    └──────────────┘    └──────┬───────┘
                                               │
                                               ▼
┌─────────────────────────────────────────────────────────┐
│                    核心处理流水线                        │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────┐│
│  │   ASR    │──▶│   LLM    │──▶│   TTS    │──▶│ 播放 ││
│  │Paraformer│   │ DeepSeek │   │ Edge-TTS │   │aplay ││
│  └──────────┘   └──────────┘   └──────────┘   └──────┘│
└─────────────────────────────────────────────────────────┘
```

### 数据流

```
[用户说话] → [麦克风] → [音频缓冲] → [VAD检测]
    ↓
[语音片段] → [ASR识别] → [文本]
    ↓
[LLM处理] → [回复文本]
    ↓
[TTS合成] → [音频文件] → [播放器] → [扬声器]
```

## ⚠️ 风险和挑战

### 技术风险

1. **音频设备兼容性**
   - 风险：Orange Pi 音频配置复杂
   - 缓解：已有详细配置教程
   - 应对：Story 1.1 优先验证

2. **ASR 模型性能**
   - 风险：识别准确率可能不达标
   - 缓解：使用经过验证的 Paraformer 模型
   - 应对：准备备选方案（SenseVoice）

3. **系统延迟**
   - 风险：总延迟可能超过 5 秒
   - 缓解：使用异步并行处理
   - 应对：持续优化各模块性能

4. **资源占用**
   - 风险：内存或 CPU 占用过高
   - 缓解：使用 INT8 量化模型
   - 应对：考虑 NPU 加速

### 项目风险

1. **模块依赖**
   - 风险：某个模块失败影响整体进度
   - 缓解：按优先级逐步验证
   - 应对：准备降级方案

2. **时间估算**
   - 风险：实际时间可能超出预期
   - 缓解：分阶段实施，及时调整
   - 应对：优先保证核心功能
